Project 0verview

ZombieBox is a 3D Android application in which the user maintains a tower defense base from a first person perspective. The player may engage in first person shooter combat with zombies to defend their base and can earn resources to upgrade and customize their experience. Zombie attacks will occur in real time waves which happen throughout the day, or on demand as the player may request an attack by spending resources. In addition to the application there will be a website which the player may use to view their game statistics, progress, and achievements as well as compare themselves to their friends.

Applicable Standards

  • Coding Standard - In the Android App, the standard language will be C Sharp and the naming conventions will be camelcase. The Web Application will be written in Ruby on Rails with Javascript, HTML, and CSS. Ruby follows the convention of snakecase (underscore_case), Javascript will be in camelcase. HAML will be the markup language representing HTML and CSS is just CSS. Active Record and SQL will be utilized to perform database queries.

Software Life Cycle Process

The software model we are using is phased development. This is a feasible option since parts of our games are modular and we have both the website and the android app to develop. After completing different goals we can add back other functionality.

Tools and Computing Environment

  • Windows Unity and Mac IOS and Yosemite are supported Operating Systems.
  • The android app will be developed with C Sharp and the Web application will be developed with Ruby on Rails, Javascript, HTML, and CSS.
  • Libraries:
  • Unity: .Net and Unity custom libraries
  • Ruby on Rails: Rails gems, Active Record, SQL
  • Javascript: JQuery, Angular.js, Three.js, Coffeescript, Underscore, Backbone.js

Configuration Management

Our group is using GitHub for version control.  Sam Davis has the repository on his GitHub account.  All group members are collaborators on the repository and are able to push changes.

Quality Assurance

When one of us completes a component, it will be reviewed by the other person working in that group. In addition to this validation, we will allow friends to Alpha test our application throughout the development process to ensure no bugs or nuances are overlooked from a developer perspective.

Risk Management

The only risk is that this game will have flashing lights which can be an issue to people prone to photosensitive seizures.  To mitigate this risk, we will display a warning when the game launches.

Table of Work Packages, Time Estimates, and Assignments

Android App - (Peter and Eric)

        Peter and Eric will be responsible for allowing users to play the game as well as interact with everything the game has to offer.

  1. Allow user to select weapon to use throughout the waves, this occurs before first wave and we will allow for users to switch between weapons during the gameplay - 9/30/15
  2. Waves of cube zombies will attack with or without the player, tower defense will play out while the player is not present, and the player may fight alongside their defenses if they are present during an attack. - 10/7/15
  3. A user must be able to move around in the game, fighting zombies and defending the base.  - 9/9/15: Peter and Eric
  4. Set up a resource collection method for which users can track progress and states throughout the game - 10/14/15
  5. Set up a game score system for which users can purchase weapons and resources in the game - 10/14/15
  6. Allow user level flexibility and difficulty variability with enemies and levels - 10/30/15
  7. Design and implement structures.

Web Application - (Sam and Tyler)

        Sam and Tyler will be responsible for completing these tasks for the web application, managing the database and model structure, and ensuring the application is hosted properly online.

  4. Users must be able to send confirmed friends messages - 10/14/15

Plan for tracking, control, and reporting of progress

Each team member will distribute their individual time spent and activity log, issues and problems, tasks completed/progress.

        After each member has submitted their progress, the web manager and unity manager will first read and analyze the data as well as examine the technical content and code that was pushed to the corresponding repositories. After an initial examination has been done, the group will meet and discuss where the project is at as well as analyzing the goals.

The group, after everything has been analyzed and discussed, will issue a Project Management Report as indicated in the deliverables section above. A Project Management Report MUST be completed AT LEAST every two weeks and will include the following information: 1-2 sentence description of the project’s status and tasks, 1-2 sentences of planned changes to the project, 1-4 sentences of each member’s thoughts and feelings on the progress and state of the project, and updated PERT chart.
